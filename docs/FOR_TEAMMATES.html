RAG Pipeline — How We Retrieve Clinical Knowledge

Five-stage retrieval pipeline. This is state-of-the-art RAG, not a naive vector search.

1
Query Rewriting

LLM transforms raw data "ferritin 12, MCV 78" into clinical query "Microcytic anemia with depleted iron — differential: iron deficiency vs ACD"

2
Metadata-Filtered Hybrid Search

Pre-filter by organ system (e.g., only hematologic chunks) → semantic + keyword search within filtered set

3
Cross-Encoder Re-Ranking

Small second model (~90MB) reads each (query, chunk) pair and scores relevance directly — much more accurate than embedding similarity

4
Citation Tracking

Every chunk carries immutable source metadata — Wallach chapter, page, pattern name

5
LLM Synthesis

Top-5 chunks → LLM generates structured explanation with citations. Every claim traceable.

Naive RAG (what most apps would do)Our RAG
Embed raw biomarker names → search → hope for the best Rewrite → filter → hybrid search → cross-encoder re-rank → synthesize
LLM generates explanation from training data → might be plausible-sounding fiction LLM receives retrieved textbook chapters → synthesizes citing specific sources
No way to verify claims Every assertion traceable to a specific chunk in the knowledge base

FAQ

Why local AI instead of calling GPT-4?
Privacy is the feature. "Your blood test never leaves your device" is a powerful claim judges understand. Also, a 1.7B medical-specific model now matches GPT-4 on clinical benchmarks — the tech is ready.
What if the local LLM is too slow?
It won't be. QVAC MedPsy 1.7B runs at ~22 tokens/sec on laptop hardware. A full synthesis takes 10–15 seconds. For a hackathon demo, that's fast enough.
What if we don't have Wallach's book?
We need excerpts for the knowledge base. The medical student handles sourcing. For the demo, we need ~30 clinical pattern descriptions covering the most common blood test findings.
What if the wearable API is a pain?
Mock data fallback. We pre-generate realistic 30-day datasets for our demo scenarios. The real API integration is bonus points.
Which module should I pick?
Frontend person → Module 6 (Dashboard). Backend/ML person → Module 4 (RAG). CV/regex person → Module 1 (OCR). Full-stack → Modules 2+3. Prompt engineer → Module 5.
What if OCR fails on Hebrew reports?
Focus on English reports for the demo. Hebrew is a stretch goal. Tesseract supports it, but right-to-left text is harder to parse correctly.
